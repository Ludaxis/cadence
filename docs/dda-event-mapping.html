Cadence DDA — Event Mapping & Priority Matrix

This document maps analytics events to Cadence DDA signal tiers, prioritized by impact on difficulty adaptation. Only P0–P2 events are required for a functional DDA pipeline. P3 events are optional enrichment.

Signal Tier Reference

Tier Name Purpose Example
BlueTier 0 Decision Quality Was the player's action good or bad? move.optimal, move.waste, progress.delta
GreenTier 1 Behavioral Tempo How fast is the player acting? tempo.interval, tempo.hesitation
YellowTier 2 Strategic Pattern What strategy is the player using? strategy.powerup, resource.efficiency
GreyTier 3 Retry & Meta Session-level retry and abandonment behavior meta.attempt, meta.session_gap, meta.abandoned
GreyTier 4 Raw Input Low-level input accuracy input.accuracy, input.rejected

These fire during active gameplay and feed the flow detector's real-time state classification (Flow, Boredom, Anxiety, Frustration).

count — times shown in this session 		The player hit a wall — out of lives. The progress value at death is a direct frustration indicator. count > 1 = repeated deaths in one session.

count — times in this session 		Player chose to continue despite failure. Willingness to invest (ad/coins) signals engagement despite difficulty.

P2 — Between-Session Enrichment BlueMEDIUM

These enrich the player model between sessions and feed StreakDamperRule and FrustrationReliefRule.
Analytics Event Cadence Signal Tier Parameters Used by DDA Why It Matters
song_start.attempt meta.attempt GreyTier 3 attempt — cumulative count High attempt count on a single level = player is stuck. Direct input to FrustrationReliefRule.
song_start.play_type meta.play_type GreyTier 3 play_type — start | restart | replay Distinguishes fresh play from retries. restart = failed and retrying. replay = mastered, replaying for fun (skip DDA adjustment).
song_booster_click strategy.powerup_attempt YellowTier 2 booster_name — hint | bomb | magnet
requirement — coin | ad | stock 		Booster tap that may not succeed (can't afford). requirement = coin when player is broke = economy pressure signal.
level_streak_update meta.streak GreyTier 3 milestone — 1 | 2 | 3
status — active | reset 		Win streak milestones and resets. Directly feeds StreakDamperRule to prevent overcorrection during hot/cold streaks.
song_result (booster counts) strategy.booster_total YellowTier 2 hint_used — count
bomb_used — count
magnet_used — count 		Total boosters consumed in a session. High total = player needed heavy assistance. Zero = player relied on skill alone.
song_result.playtime tempo.session_duration GreenTier 1 playtime — seconds Session duration relative to expected level length. Abnormally long = struggling. Abnormally short = too easy or gave up early.

P3 — Economy & Profile Context GreenLOW

Optional enrichment signals. These don't fire during gameplay but provide cross-session context for the player model. Implement these after P0–P2 are validated.
Analytics Event Cadence Signal Tier Parameters Used by DDA Why It Matters
item_earned resource.earned YellowTier 2 item_id — currency type
amount — quantity
source — common_level_reward | daily_level_reward | treasure_hunt | gold_tile 		Tracks resource income. Low earn rate + high spend = economy squeeze, which indirectly affects difficulty tolerance.
item_spent resource.spent YellowTier 2 item_id — currency type
amount — quantity
source — hint | bomb | magnet 		Tracks spend on boosters. Spend rate rising over time = player is compensating for difficulty with economy.
booster_update resource.stored YellowTier 2 booster_name — hint | bomb | magnet | infinite_lives
amount — quantity 		Inventory snapshot. Empty inventory + high difficulty = player has no safety net.
User property: total_level_played (profile enrichment) — total_level_played — integer Player maturity indicator. More levels played = more data for Glicko-2 confidence.
User property: coin_balance (profile enrichment) — coin_balance — integer Economy health check. Near-zero balance constrains booster access, effectively raising difficulty.
me_start / me_result (duplicate filtering) — Same as song_start / song_result Only fires if playtime ≥ 15s. Can be used to filter out accidental starts or instant quits before feeding DDA.

P4 — Not Needed for DDA GreySKIP

These events serve analytics, monetization, or UI tracking purposes. They carry no gameplay performance signal and should not be wired to Cadence.
Category Events Reason to Skip
UI Navigation song_impression, song_click, song_ap, screen_open, popup_open, button_click Browsing and navigation behavior — no gameplay skill signal
Ad Monetization fullads_request, fullads_show, rewarded_*, ad_impression, all Firebase/AppsFlyer ad events Business metrics — ad exposure does not indicate player skill
IAP iap_impression, iap_click, iap_purchased Purchase behavior — monetization tracking, not performance
Subscriptions sub_cata_show, sub_cata_click, sub_purchased Subscription funnel — business metrics only
FTUE ftue_main (step_01 through step_28) Tutorial tracking — DDA should be disabled entirely during onboarding
Collection artwork_unlock, artwork_click Meta-game progression — no difficulty signal
Cosmetics avatar_update, frame_update Cosmetic customization — irrelevant to difficulty
Treasure Hunt treasure_hunt_update, treasure_hunt_reward Meta-game progress — does not reflect in-level performance
Account login, all ULS events Authentication and social — no gameplay data
Content Metadata All ACM events Content catalog tagging — infrastructure, not gameplay
